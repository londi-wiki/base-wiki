## Handlungsziel 1 – Anforderungen analysieren und deklarativ beschreiben

> *Analysiert und beschreibt Anforderungen, so dass eine Realisierung in funktionaler Programmierung erfolgen kann.*

### 1.1 Imperativ vs. Deklarativ/Funktional

Beim **imperativen** Paradigma beschreibt man *wie* etwas gemacht wird (Schritt für Schritt). Beim **deklarativen/funktionalen** Paradigma beschreibt man *was* das Ergebnis sein soll.

```java
// Imperativ: WIE wird summiert?
int sum = 0;
for (int i = 0; i < numbers.size(); i++) {
    sum += numbers.get(i);
}

// Deklarativ/Funktional: WAS soll passieren?
int sum = numbers.stream().reduce(0, Integer::sum);
```

Der deklarative Ansatz abstrahiert die Kontrollstruktur (Loop, Index, Akkumulator) weg und drückt nur die Absicht aus.

### 1.2 Deklarative Beschreibung von Problemen und Endzuständen

Beim deklarativen Ansatz formuliert man das Problem als Transformation von Input zu Output, ohne sich um Zwischenschritte zu kümmern.

**Beispiel:** "Gib mir alle volljährigen Personen, sortiert nach Name."

```java
// Deklarativ: Beschreibe den gewünschten Endzustand
List<Person> result = personen.stream()
    .filter(p -> p.alter() >= 18)
    .sorted(Comparator.comparing(Person::name))
    .toList();
```

Typische Vorgehensweisen:
- **Problem als Datentransformation** beschreiben (Input → Output)
- **Zwischenzustände** als Pipeline-Schritte modellieren (filter → map → collect)
- **Endzustand** definieren statt Kontrollfluss festlegen

### 1.3 Vor- und Nachteile funktionaler Programmierung

**Vorteile:**
- Bessere Testbarkeit durch Pure Functions (gleicher Input = gleicher Output)
- Weniger Bugs durch Immutability (kein unerwarteter State-Change)
- Einfachere Parallelisierung (keine shared mutable state)
- Kompakterer, lesbarerer Code bei Datentransformationen
- Einfacheres Reasoning — man kann Funktionen isoliert betrachten

**Nachteile:**
- Steilere Lernkurve, besonders in primär OOP-Sprachen wie Java
- Performance-Overhead bei vielen kurzlebigen Objekten (z.B. Streams erzeugen Wrapper)
- Debugging kann schwieriger sein (Stack-Traces bei verschachtelten Lambdas)
- Nicht alles lässt sich elegant funktional lösen (z.B. I/O, GUI, State-Management)

### 1.4 Wichtige Begriffe

| Begriff | Erklärung | Java-Beispiel |
|---|---|---|
| **Pure Function** | Keine Seiteneffekte, gleicher Input → gleicher Output | `Function<Integer, Integer> dbl = x -> x * 2;` |
| **Immutable Data** | Daten, die nach Erstellung nicht verändert werden | `List.of(1, 2, 3)`, `record Point(int x, int y){}` |
| **Lambda-Expression** | Anonyme Funktion als Wert | `(a, b) -> a + b` |
| **Function** | Wert, der als Parameter übergeben/zurückgegeben werden kann | `Function<String, Integer> len = String::length;` |
| **Closure** | Lambda, die Variablen aus dem umgebenden Scope einfängt | `int factor = 3; Function<Integer,Integer> f = x -> x * factor;` |
| **Callback** | Funktion, die einer anderen Funktion als Argument übergeben wird | `list.forEach(System.out::println);` |
| **Functor** | Struktur, auf die man `map` anwenden kann | `Optional`, `Stream` |

```java
// Closure: Die Variable 'greeting' wird eingefangen
String greeting = "Hallo";
Function<String, String> greet = name -> greeting + " " + name;
greet.apply("Welt"); // "Hallo Welt"
```

### 1.5 Realisierungskonzept erarbeiten

Bevor man funktional implementiert, sollte man sich überlegen:

1. **Datenmodell definieren** — Was sind die Input-/Output-Typen? → Records, Immutable Classes
2. **Transformationen identifizieren** — Welche Schritte braucht es von Input zu Output?
3. **Funktionale Bausteine wählen** — Welche Funktionen (map, filter, reduce, etc.) passen?
4. **Seiteneffekte isolieren** — I/O und State-Änderungen an den Rand der Architektur schieben

```java
// Konzept: Bestellungen → Umsatz pro Kategorie
// 1. Input: List<Order>
// 2. Transformationen: filtern (nur bezahlt) → gruppieren (nach Kategorie) → summieren
// 3. Output: Map<String, Double>

record Order(String kategorie, double betrag, boolean bezahlt) {}

Map<String, Double> umsatzProKategorie = orders.stream()
    .filter(Order::bezahlt)
    .collect(Collectors.groupingBy(
        Order::kategorie,
        Collectors.summingDouble(Order::betrag)
    ));
```

## Handlungsziel 2 – Funktional implementieren

> *Implementiert Algorithmen und Teilprobleme von Applikationen effizient nach dem funktionalen Programmierparadigma gemäss vorgegebenen Anforderungen.*

### 2.1 Funktionalen Code lesen, verstehen, warten und erweitern

Funktionaler Code in Java basiert auf der **Streams API** und **Functional Interfaces**. Schlüssel zum Verständnis ist, die Pipeline von links nach rechts zu lesen:

```java
// Lese: "Nimm Strings, filtere die langen, mach sie gross, sortiere, sammle als Liste"
List<String> result = strings.stream()
    .filter(s -> s.length() > 3)
    .map(String::toUpperCase)
    .sorted()
    .toList();
```

**Wartbarkeit verbessern** durch benannte Funktionen:

```java
Predicate<String> istLangGenug = s -> s.length() > 3;
Function<String, String> normalisiere = String::toUpperCase;

List<String> result = strings.stream()
    .filter(istLangGenug)
    .map(normalisiere)
    .sorted()
    .toList();
```

### 2.2 Funktionale Programmierkonzepte (Filter-Map-Reduce) und Entwurfsmuster

#### Filter-Map-Reduce

Das ist das Brot-und-Butter-Pattern der funktionalen Programmierung:

```java
List<Product> products = List.of(
    new Product("Laptop", 1200.0, "Elektronik"),
    new Product("Buch", 25.0, "Bücher"),
    new Product("Maus", 45.0, "Elektronik"),
    new Product("Roman", 18.0, "Bücher")
);

// Filter → Map → Reduce: Gesamtpreis aller Elektronik-Produkte
double totalElektronik = products.stream()
    .filter(p -> p.kategorie().equals("Elektronik"))  // FILTER
    .map(Product::preis)                                // MAP
    .reduce(0.0, Double::sum);                          // REDUCE
// Ergebnis: 1245.0
```

#### Builder-Pattern (funktionaler Stil)

Das Builder-Pattern passt gut zum funktionalen Stil, weil jeder Methodenaufruf ein neues/modifiziertes Objekt zurückgibt:

```java
public record HttpRequest(String url, String method, Map<String, String> headers) {

    public static HttpRequest get(String url) {
        return new HttpRequest(url, "GET", Map.of());
    }

    public HttpRequest withHeader(String key, String value) {
        var newHeaders = new HashMap<>(headers);
        newHeaders.put(key, value);
        return new HttpRequest(url, method, Map.copyOf(newHeaders));
    }
}

// Verwendung: Jeder Schritt erzeugt ein neues, unveränderliches Objekt
HttpRequest request = HttpRequest.get("https://api.example.com")
    .withHeader("Authorization", "Bearer token123")
    .withHeader("Accept", "application/json");
```

### 2.3 Funktionalen Code verteilt, parallel und nebenläufig ausführen

Java Streams unterstützen Parallelisierung direkt:

```java
// Sequentiell
long count = numbers.stream()
    .filter(n -> isPrime(n))
    .count();

// Parallel — nutzt ForkJoinPool automatisch
long count = numbers.parallelStream()
    .filter(n -> isPrime(n))
    .count();
```

**Wann parallel Streams nutzen?**
- Grosse Datenmengen (tausende+ Elemente)
- CPU-intensive Operationen pro Element
- Keine Seiteneffekte in den Operationen!

**CompletableFuture** für asynchrone funktionale Pipelines:

```java
CompletableFuture<String> result = CompletableFuture
    .supplyAsync(() -> fetchDataFromApi())        // Async starten
    .thenApply(data -> parseJson(data))           // Transformieren
    .thenApply(parsed -> extractRelevant(parsed)) // Weiter transformieren
    .exceptionally(ex -> "Fehler: " + ex.getMessage());
```

> **Achtung:** Parallel Streams sind kein Allheilmittel. Bei kleinen Datenmengen oder I/O-lastigen Operationen ist die sequentielle Variante oft schneller, weil der Overhead für Thread-Koordination entfällt. Das ist kein Meinungsding — das lässt sich messen.

## Handlungsziel 3 – Imperativen Code funktional refactoren

> *Verbessert und optimiert bestehenden imperativ implementierten Code durch Anwendung von funktionaler Programmierung (Refactoring).*

### 3.1 Programmteile analysieren und funktional optimieren

**Erkennungsmerkmale** für Code, der vom funktionalen Refactoring profitiert:
- Schleifen mit Akkumulatoren (`for` + temporäre Variable)
- Verschachtelte `if`-Statements zur Datenfilterung
- Manuelle Null-Checks
- Mutable Collections, die schrittweise aufgebaut werden

**Vorher (imperativ):**

```java
List<String> result = new ArrayList<>();
for (Person p : personen) {
    if (p.alter() >= 18) {
        String name = p.name().toUpperCase();
        if (!result.contains(name)) {
            result.add(name);
        }
    }
}
Collections.sort(result);
```

**Nachher (funktional):**

```java
List<String> result = personen.stream()
    .filter(p -> p.alter() >= 18)
    .map(p -> p.name().toUpperCase())
    .distinct()
    .sorted()
    .toList();
```

Der funktionale Code ist kürzer, klarer in der Absicht, und eliminiert mutable State.

### 3.2 Funktionale Elemente mit imperativen/deklarativen Paradigmen kombinieren

In der Praxis verwendet man in Java fast immer einen **Mix** aus OOP und funktionalem Stil. Die Kunst liegt darin, für jede Situation den passenden Stil zu wählen.

```java
// Klasse (OOP) mit funktionalen Methoden
public class OrderService {
    private final List<Order> orders; // State (OOP)

    public OrderService(List<Order> orders) {
        this.orders = List.copyOf(orders); // Immutable (funktional)
    }

    // Funktionale Datenverarbeitung
    public Map<String, DoubleSummaryStatistics> statistikProKategorie() {
        return orders.stream()
            .collect(Collectors.groupingBy(
                Order::kategorie,
                Collectors.summarizingDouble(Order::betrag)
            ));
    }

    // Kombination: Imperativ (Logging/Side-Effect) + Funktional (Transformation)
    public List<Order> verarbeite(Predicate<Order> filter) {
        List<Order> gefiltert = orders.stream()
            .filter(filter)
            .toList();

        System.out.println("Verarbeite " + gefiltert.size() + " Bestellungen");
        return gefiltert;
    }
}
```

**Faustregel:** Datenverarbeitung → funktional. State-Management, I/O, Lifecycle → OOP/imperativ. Die Seiteneffekte werden an den Rand der Applikation geschoben, der Kern ist rein funktional.

## Handlungsziel 4 – Korrektheit und Qualität überprüfen

> *Überprüft Implementierung auf Korrektheit und Qualität.*

### 4.1 Funktionalen Code testen (Unit-Tests)

Pure Functions sind ein Traum zum Testen — kein Setup, kein Mocking, einfach Input rein, Output prüfen:

```java
class MathUtilsTest {

    // Pure Function testen — super einfach
    @Test
    void testFilter() {
        List<Integer> input = List.of(1, 2, 3, 4, 5, 6);
        Predicate<Integer> istGerade = n -> n % 2 == 0;

        List<Integer> result = input.stream()
            .filter(istGerade)
            .toList();

        assertEquals(List.of(2, 4, 6), result);
    }

    // Funktionale Pipeline testen
    @Test
    void testPipeline() {
        Function<List<String>, List<String>> pipeline = names -> names.stream()
            .filter(n -> n.length() > 3)
            .map(String::toUpperCase)
            .sorted()
            .toList();

        assertEquals(
            List.of("ALICE", "CLARA"),
            pipeline.apply(List.of("Bob", "Alice", "Jo", "Clara"))
        );
    }

    // Higher-Order Function testen
    @Test
    void testHigherOrderFunction() {
        Function<Integer, Function<Integer, Integer>> multiplier =
            factor -> number -> factor * number;

        Function<Integer, Integer> triple = multiplier.apply(3);
        assertEquals(15, triple.apply(5));
        assertEquals(9, triple.apply(3));
    }
}
```

**Property-Based Testing** ist besonders gut geeignet für funktionalen Code. Statt einzelne Werte zu testen, testet man Eigenschaften:

```java
// Konzeptionell (z.B. mit jqwik):
// "Für jede Liste gilt: nach sortieren ist jedes Element >= dem vorherigen"
@Property
void sortierteListeIstSortiert(@ForAll List<Integer> input) {
    List<Integer> sorted = input.stream().sorted().toList();
    for (int i = 1; i < sorted.size(); i++) {
        assertTrue(sorted.get(i) >= sorted.get(i - 1));
    }
}
```

### 4.2 Coderichtlinien einhalten

Wichtige Konventionen für funktionalen Java-Code:

- **Lambdas kurz halten** — Mehr als 3 Zeilen? → Eigene Methode extrahieren
- **Aussagekräftige Variablennamen** für Functional Interfaces (`istGueltig` statt `p`)
- **Keine Seiteneffekte in Stream-Operationen** — `peek()` nur zum Debugging
- **`var` sparsam nutzen** — Bei komplexen Generics lieber explizite Typen
- **Streams nicht wiederverwenden** — Ein Stream kann nur einmal konsumiert werden!

```java
// ❌ Schlecht: Seiteneffekt in map
List<String> sideEffects = new ArrayList<>();
stream.map(x -> {
    sideEffects.add(x); // NEIN!
    return x.toUpperCase();
});

// ✅ Gut: Reine Transformation
stream.map(String::toUpperCase).toList();
```

### 4.3 Best-Practices für funktionale Programmierung

1. **Bevorzuge Immutability:** `List.of()`, `Map.of()`, Records, `Collections.unmodifiable*()`
2. **Vermeide `null`:** Nutze `Optional` als Rückgabetyp
3. **Compose small functions:** Kleine, wiederverwendbare Funktionen zusammenstecken
4. **Keine verschachtelten Streams:** Lieber mehrere Schritte oder `flatMap`
5. **Seiteneffekte isolieren:** Pure Core, impure Shell (Functional Core, Imperative Shell)

```java
// Best-Practice: Functional Core, Imperative Shell

// CORE — rein funktional, leicht testbar
record Discount(double percentage) {
    double apply(double price) {
        return price * (1 - percentage / 100);
    }
}

Function<List<Double>, Double> calculateTotal = prices -> prices.stream()
    .map(new Discount(10)::apply)
    .reduce(0.0, Double::sum);

// SHELL — hier passieren die Seiteneffekte
public void processOrder(List<Double> prices) {
    double total = calculateTotal.apply(prices); // Pure
    database.save(total);                        // Seiteneffekt
    emailService.sendReceipt(total);             // Seiteneffekt
}
```
